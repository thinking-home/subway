# ThinkingHome.DeviceModel

Нейтральная модель устройств умного дома и протокол взаимодействия с ними.

Это слой абстракции, **не зависящий ни от конкретной экосистемы** (Яндекс Алиса, Matter,
Apple HomeKit), **ни от конкретного транспорта устройств** (NooLite, MQTT/Zigbee, Waterius, …).
На него сверху ставятся адаптеры экосистем, снизу — драйверы устройств.

## Зачем это нужно

Изначально модель Алисы (`ThinkingHome.Alice`) совмещала две разные роли: *описание устройства*
и *формат обмена с Яндексом*. Устройство само собирало JSON Алисы (см. `TestBulb.GetDescription()`).
Это приводит к трём проблемам:

1. **Нет нейтрального слоя** — не с чем сверять совместимость с Matter и другими экосистемами.
2. **Взрыв N×M** — при добавлении Matter/HomeKit каждое устройство обязано реализовать ещё и
   `GetMatterDescription()`, `GetHomeKitDescription()` и т.д.
3. **Логика устройства связана с транспортом ассистента** — переносимость и тестируемость страдают.

`ThinkingHome.DeviceModel` вводит один нейтральный контракт устройства. Драйверы реализуют только его,
адаптеры экосистем только читают его и переводят в свой формат. Устройство ничего не знает про Алису.

## Архитектура

```
   Яндекс Алиса        Matter          Apple HomeKit          ← экосистемы
        │                │                   │
  AliceMapper       MatterMapper       HomeKitMapper           ← АДАПТЕРЫ (перевод нейтральной
        │                │                   │                   модели в формат экосистемы)
        └────────────────┼───────────────────┘
                         │
              ThinkingHome.DeviceModel                             ← ЯДРО: нейтральная модель +
              (модель + протокол + реестр)                       протокол + реестр устройств
                         │
        ┌────────────────┼───────────────────┐
     NooLite      MQTT / Zigbee2MQTT      Waterius              ← ДРАЙВЕРЫ (реализуют
        │                │                   │                    нейтральный контракт устройства)
     железо           брокер              счётчики
```

`ThinkingHome.Alice` в этой схеме — **один из адаптеров** (первый). Его DTO-классы остаются как есть,
но становятся *выходом маппера*, а не *входом устройства*.

## Модель данных

Иерархия намеренно повторяет модель данных Matter (см. раздел «Протокол Matter» ниже), чтобы
совместимость проверялась структурно, а не подгонялась потом:

- **Device (Node)** — устройство целиком: стабильный `Id` (переживает рестарты), метаданные
  (производитель, модель, версии), комната.
- **Endpoint** — логическая часть устройства. Одно физическое устройство может нести несколько
  endpoint'ов (например, двухклавишный выключатель = 2 endpoint'а с `OnOff`). У endpoint'а есть
  тип (`DeviceType`) и наборы способностей и свойств.
- **Capability** — способность, которой можно **управлять** (актуатор): `OnOff`, `Level` (яркость,
  громкость), `Color`, `Mode`, `Range`, `Toggle`. Несёт параметры (единицы, диапазон, шаг).
- **Property** — свойство, которое можно только **читать** (сенсор/событие): температура, влажность,
  движение, открытие, заряд батареи.
- **State** — текущее значение способностей и свойств.
- **Command** — запрос на изменение состояния способности.

> Adapter'у Алисы (плоская модель `capabilities[]`) endpoint'ы **схлопываются**;
> Matter-адаптеру они отдаются 1:1. Поэтому endpoint'ы заложены в модель сразу — ретрофитить
> их потом дорого.

### Дискриминация типов: закрытые иерархии + pattern matching

Способность конкретна (у `OnOff` значение `bool`, у `Level` — `int`, у `Color` — `hsv`), а хост
оперирует общими типами. Нативный .NET-эквивалент полиморфной десериализации Алисы — **закрытая
(sealed) иерархия типов + сопоставление с образцом**: конкретный тип заменяет строку-дискриминатор,
`switch` заменяет выбор типа десериализатором. Значение несёт свой родной тип, а не `object`.

Три параллельные закрытые иерархии (описание / команда / состояние) с ключом-способностью:

```csharp
public abstract record DeviceCommand { public required string Instance { get; init; } }
public sealed record OnOffCommand : DeviceCommand { public required bool Value { get; init; } }
public sealed record LevelCommand : DeviceCommand { public required int  Value { get; init; } } // 0..100
// ... ColorCommand, ModeCommand, RangeCommand
```

Превращение «общее → конкретное» происходит в трёх местах, и все сходятся на одной иерархии:

1. **Фасад** строит конкретный тип: `new OnOffCommand { Instance = "on", Value = true }`.
2. **Драйвер** разбирает `switch`'ем — это и есть дискриминатор; незнакомая команда →
   `CommandOutcome.Unsupported` (→ ошибка Алисы `INVALID_ACTION`).
3. **Маппер Алисы** — его полиморфная десериализация сохраняется, но теперь выдаёт нейтральный тип
   (`OnOffCommand`) вместо DTO Алисы. Внутри ядра дискриминатор — сам .NET-тип, а не строка.

Закрытость оправдана: словарь способностей = конечный набор кластеров Matter, новая способность —
осознанное добавление. Нужна компайл-тайм полнота обработки — вместо `switch` берётся visitor.

## Протокол взаимодействия

Всего четыре примитива. Они изоморфны тому, что требуют и Яндекс, и Matter, и HomeKit:

| Примитив      | Направление            | Смысл                          | Яндекс                     | Matter               |
|---------------|------------------------|--------------------------------|----------------------------|----------------------|
| **Discovery** | драйвер → реестр        | зарегистрировать устройство    | `GET /user/devices`        | enum узлов/endpoint'ов |
| **Query**     | адаптер → драйвер       | опросить текущее состояние     | `POST /user/devices/query` | Read attribute       |
| **Execute**   | адаптер → драйвер       | исполнить команду              | `POST /user/devices/action`| Invoke command       |
| **Report**    | драйвер → реестр → адаптер | push об изменении состояния | callback state notification| Subscribe / Report   |

Правила, без которых интеграция не работает:

1. **Стабильный `Id`** — переживает рестарты и переподключения. Иначе ассистент не свяжет устройство.
2. **Достоверный результат команды** — `Execute` возвращает исход (`Done` / `Error` + код), а не
   «выстрелил и забыл». Односторонние транспорты без ACK (например, NooLite на передачу) честно
   помечаются как `Reportable = false` / best-effort.
3. **Нормализация значений** — ядро оперирует нормализованными единицами (проценты, °C, кельвины
   для цвета). Перевод в единицы экосистемы (упаковка hsv в int у Алисы, 0..255 у NooLite) живёт
   в адаптерах и драйверах, не в ядре.
4. **Push, а не polling** — изменения приходят событием `Report` и фанятся подписчикам.
5. **Двухуровневый кэш состояния** — хост хранит последнее отрепорченное состояние каждого
   устройства и отвечает `Query` из него мгновенно (у Алисы жёсткий таймаут), а поход в драйвер —
   фолбэк. Драйвер, в свою очередь, может держать свой кэш и/или периодически опрашивать железо.

## Публичный .NET API

Потребитель-грань хоста — это и есть публичный .NET API слоя устройств. Он **нативный**
(in-process, без сериализации и маппинга) и живёт внутри `ThinkingHome.DeviceModel`. Все внешние
адаптеры (`AliceMapper` и будущие Matter/HomeKit) — тоже его потребители: они не лезут в реестр
напрямую, а вызывают этот API, добавляя лишь перевод форматов. То есть .NET API — ствол, адаптеры
экосистем — ветки на нём.

У хоста две грани, намеренно разделённые (у драйвера и у потребителя разные права и смысл):

```csharp
// провайдер-грань — реализуют драйверы
public interface IDeviceRegistry {
    IDisposable Register(IDevice device);   // Dispose() = Unregister
}

// потребитель-грань — это и есть .NET API; его зовут и мапперы, и ThinkingHome
public interface IDeviceHost {
    IReadOnlyCollection<DeviceDescriptor> GetDevices();
    Task<DeviceSnapshot> QueryAsync(string deviceId, CancellationToken ct = default);
    Task<CommandOutcome> ExecuteAsync(string deviceId, DeviceCommand cmd, CancellationToken ct = default);
    event Action<StateChange> Changed;      // агрегированный поток изменений всех устройств
}
```

Полнота и удобство — разные цели, поэтому API делается двухслойным:

- **Минимальное ядро** (`IDeviceHost` / `IDeviceRegistry` / `IDevice`) — маленькое, ортогональное,
  стабильное. Его реализуют хост и драйверы, на него опираются адаптеры. Оптимизировано на
  корректность и стабильность контракта.
- **Эргономичный фасад** — тонкий слой поверх ядра для разработчика (сценарии, встраивание в
  ThinkingHome): типобезопасные хендлы устройств, типизированный доступ к способностям,
  fluent-команды, подписки. Оптимизирован на удобство; меняется, не дестабилизируя ядро.

Разница на примере — одна и та же команда:

```csharp
// через ядро (обобщённо; так пользуются адаптеры):
await host.ExecuteAsync("noolite:ch5", new OnOffCommand { Instance = "on", Value = true });

// через фасад (так пользуется разработчик .NET):
await host.Device("noolite:ch5").OnOff().TurnOnAsync();

var light = host.Device("noolite:ch5");
if (light.Level() is { } level) await level.SetAsync(50);   // способности нет → null
light.OnChanged(change => Log(change));                     // подписка на изменения
```

Фасад — это сахар (в основном extension-методы) над ядром, поэтому ядро остаётся маленьким.

# Протокол Matter (краткая выжимка)

[Matter](https://csa-iot.org/all-solutions/matter/) — открытый стандарт межвендорной совместимости
устройств умного дома от Connectivity Standards Alliance (CSA), бывший Project CHIP. Версия 1.0 —
октябрь 2022. Работает поверх IP (Wi-Fi, Thread, Ethernet; BLE — только для первичной привязки).
Поддержан Apple, Google, Amazon, Samsung и др. Цель — одно устройство работает во всех экосистемах.

## Что мы берём из Matter, а что нет

Matter — это два слоя: **модель данных** (что такое устройство и его функции) и **транспортный стек**
(как устройства находят друг друга и защищённо общаются).

- **Берём:** модель данных и семантику операций (interaction model) — как *словарь и структуру*
  нашей нейтральной модели.
- **Не берём:** транспорт (Thread/Wi-Fi/BLE), привязку (commissioning: PASE/CASE), fabrics,
  сертификаты устройств, TLV-кодирование. Физически с Алисой мы общаемся по HTTP, а не по Matter.

То есть Matter для нас — это **эталонный словарь и структура**, обеспечивающие будущую совместимость,
а не сетевой стек, который надо реализовать.

## Модель данных

Иерархия сверху вниз:

- **Node (узел)** — логическое устройство в сети, имеет IP и уникальные идентификаторы.
- **Endpoint (точка подключения)** — нумерованная часть узла. Endpoint `0` — служебный (корневой,
  описывает сам узел). Каждый прикладной endpoint представляет одно логическое под-устройство и
  имеет **Device Type**.
- **Cluster (кластер)** — функциональный блок endpoint'а с собственным id. Имеет роль:
  *server* (хранит состояние, отвечает на команды) или *client*. Пример: `On/Off`, `Level Control`.
- **Attribute (атрибут)** — единица состояния кластера. Читается, подписывается, иногда записывается.
- **Command (команда)** — действие над кластером; может иметь параметры и ответ.
- **Event (событие)** — запись в журнал кластера (например, сработал датчик).
- **Device Type (тип устройства)** — задаёт обязательный набор кластеров для endpoint'а
  (например, «Dimmable Light» обязан иметь `On/Off` и `Level Control`).

## Interaction Model — операции

Как взаимодействуют узлы (независимо от транспорта):

- **Read** — прочитать значения атрибутов.
- **Write** — записать атрибут.
- **Invoke** — вызвать команду кластера.
- **Subscribe** — подписаться на изменения атрибутов/событий, получать **Report** (push-отчёты).

## Машиночитаемый словарь и валидация полноты

Модель данных Matter доступна в открытом виде (репозиторий `project-chip/connectedhomeip`,
Apache-2.0) — можно автоматически сверять полноту нашего словаря и в перспективе генерировать типы.
Два источника:

- **`data_model/<версия>/`** (`clusters/`, `device_types/`) — авторитетное, спек-точное,
  версионированное XML модели данных. Ведёт Data Model Tiger Team CSA, обновляется с каждым релизом
  спеки, используется для сертификации. **Наш выбор для валидации полноты.**
- **`src/app/zap-templates/zcl/data-model/chip/`** (`matter-devices.xml` и др.) — входы для ZAP,
  которым Matter генерирует свой C++. Прецедент и материал для будущей автогенерации.

Как применяем:

1. **Пин версии.** Vendor-snapshot или git-submodule `data_model/<версия>/` на конкретном теге —
   воспроизводимость сборки. Тянем только нужные device types (свет, розетка, термостат, датчики),
   а не весь Matter.
2. **Тест полноты (CI-гейт).** Парсим XML и для каждого нашего `DeviceType` проверяем, что
   `Capability`/`Property` покрывают обязательные кластеры и их обязательные атрибуты/команды. Тест
   падает, если в Matter есть обязательный элемент, которого у нас нет, или если мы ссылаемся на
   несуществующий id.
3. **Автогенерация (потом).** Roslyn source generator читает то же XML → эмитит нейтральные
   enum'ы/записи и каркас дескрипторов. Так ZAP делает для C++.

Границы: XML даёт только нейтральный/Matter-словарь — маппинги в Яндекс/HomeKit и нормализация единиц
(hue `0..254`, майреды, `0.01 °C` → наши %, °C, K) остаются ручными.
